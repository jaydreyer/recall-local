## SECTION 1: THE 30-SECOND PITCH (Overview Post)

### What Is Recall.local?

Recall.local is a privacy-first, local AI operations agent. It ingests messy, real-world inputs — documents, URLs, emails, meeting transcripts, bookmarks — builds searchable vector + structured memory from them, and executes reliable automations on top of that memory. The entire system runs locally on a single machine using open-source models (Ollama), with optional cloud LLM fallback (Claude, GPT-4o, Gemini) for resilience.

It is a full-stack system spanning:
- A FastAPI backend with 25+ API endpoints
- Two React dashboards (general ops + job search command center)
- A Chrome extension (Manifest V3) for browser-based capture
- An n8n workflow automation layer with scheduled pipelines
- PostgreSQL-style structured storage (SQLite) + vector memory (Qdrant)
- Docker Compose orchestration (7 containerized services)
- A multi-provider LLM abstraction layer (Ollama, Anthropic, OpenAI, Gemini)
- An evaluation harness with golden-set testing and soak runs
- Automated backup/restore with checksums
- Telegram bot integration for real-time alerts

The system was built across 7 phases, from bootstrap to a fully operational job search intelligence platform.

### Why It Exists

Jay built Recall.local as a portfolio project that proves he can design, architect, and ship a production-grade AI system end-to-end — not just call an API, but build the entire pipeline: ingestion, embedding, retrieval, generation, evaluation, observability, and operations. It showcases product management thinking (phased roadmap, PRD-driven development) alongside deep engineering execution.

## SECTION 2: THE DUAL MEMORY ARCHITECTURE

### Technical Summary

Recall.local uses a dual memory system that mirrors how humans store knowledge — a semantic/associative layer for "what does this remind me of?" and a structured/operational layer for "what happened and when?"

**Vector Memory (Qdrant):**
- 3 collections: `recall_docs` (general knowledge), `recall_jobs` (job listings), `recall_resume` (resume chunks)
- 768-dimension embeddings generated by nomic-embed-text via Ollama
- Cosine similarity search with configurable top-k and minimum score thresholds
- Supports hybrid retrieval: blending dense semantic vectors with sparse BM25-style token overlap
- Configurable alpha parameter (default 0.65 dense / 0.35 sparse) for tuning precision vs recall
- Optional heuristic reranking using Jaccard similarity on token and bigram overlap

**Structured Memory (SQLite):**
- `runs` table: Every workflow execution tracked with status, model, latency, input hash
- `ingestion_log`: Full audit trail of every document ingested — source, channel, chunks created, tags
- `eval_results`: Test outcomes with citation validity, latency, pass/fail
- `alerts`: Operational alerts tied to specific runs
- `settings`, `resume_versions`, `company_profiles`: Phase 6 operational state

**Why dual memory matters:** Vector search is great for "find me something similar" but terrible for "show me everything that happened last Tuesday" or "how many documents did I ingest this week?" The structured layer provides the operational awareness that pure vector systems lack.

### The RAG Pipeline in Detail

1. **Query embedding:** User query is embedded using nomic-embed-text (768 dims)
2. **Candidate retrieval:** Top-k candidates fetched from Qdrant (default k=5)
3. **Hybrid scoring (optional):** Dense scores blended with sparse token-overlap scores
4. **Heuristic reranking (optional):** Post-retrieval reranking using Jaccard similarity on tokens + bigrams
5. **Context assembly:** Retrieved chunks assembled into a context window
6. **Prompted generation:** LLM generates a cited answer using only the retrieved context
7. **Citation validation:** Every citation (doc_id, chunk_id) verified against actual retrieved chunks
8. **Retry on failure:** If validation fails, retry with lower temperature and stricter prompt
9. **Artifact persistence:** Final answer saved as versioned artifact in `/data/artifacts/rag/`

**Three query modes:**
- **Default:** General knowledge retrieval
- **Job Search Coach:** Frames answers as practical career coaching — mentions the user by name, uses coaching terminology (experience, role, impact, business value, fit)
- **Learning Coach:** Concise teaching language emphasizing tradeoffs and practical implications

## SECTION 3: THE MULTI-SOURCE INGESTION ENGINE

### Technical Summary

Recall.local can ingest content from virtually any source through a unified pipeline that normalizes everything into searchable, tagged, grouped chunks.

**Supported input types:**
- Plain text
- URLs (web pages, articles, documentation)
- PDF documents
- Word documents (DOCX)
- Markdown files
- HTML files
- Email messages (EML format)
- Meeting transcripts

**Ingestion channels (how content arrives):**
- **Webhook:** Direct API calls from any HTTP client
- **Chrome Extension:** Right-click context menu capture from any browser tab
- **iOS Share Sheet:** Mobile capture via share extension
- **Gmail Forward:** Email-to-ingest pipeline
- **File Upload:** Direct file upload via API or dashboard
- **Folder Watch:** Automatic ingestion of files dropped into `data/incoming/`
- **Obsidian Vault Sync:** Continuous sync from Obsidian markdown vault with hash-based deduplication

**The pipeline:**
1. **Channel adapter normalization:** Each source channel has an adapter that normalizes the payload into a canonical `IngestRequest` (source_type, content, channel, title, group, tags, metadata)
2. **Content extraction:** trafilatura for URLs, pdfplumber for PDFs, python-docx for DOCX, etc.
3. **Auto-tagging:** URL patterns, title keywords, and email sender patterns automatically assign canonical groups (job-search, learning, project, reference, meeting) and suggested tags
4. **Semantic chunking:** Content split into overlapping chunks with configurable token sizes
5. **Embedding:** Each chunk embedded via nomic-embed-text (768 dims)
6. **Vector upsert:** Chunks stored in Qdrant with rich metadata payload
7. **Audit logging:** Every ingestion logged to SQLite with source, channel, chunk count, status
8. **Optional Obsidian write-back:** Ingested content can optionally be written back to Obsidian vault

**Deduplication:** Source keys and content hashes prevent duplicate ingestion. The Obsidian vault sync uses MD5 content hashes to detect changed files.

## SECTION 4: THE LLM PROVIDER ABSTRACTION

### Technical Summary

Recall.local has a single Python module (`llm_client.py`) that abstracts away the differences between four LLM providers behind a unified interface. Switching providers is a single environment variable change.

**Supported providers:**
| Provider | Default Model | Use Case |
|----------|--------------|----------|
| Ollama (local) | llama3.2:3b | Privacy-first default, zero API cost |
| Anthropic | claude-sonnet-4-20250514 | Cloud fallback, high-quality evaluation |
| OpenAI | gpt-4o-mini | Alternative cloud option |
| Gemini | gemini-2.5-flash | Google ecosystem integration |

**Key design decisions:**
- **Single env var swap:** `RECALL_LLM_PROVIDER=ollama|anthropic|openai|gemini`
- **Retry with exponential backoff:** Configurable retries (default 3) with 1.5s backoff
- **Timeout management:** 180s for Ollama (local inference is slower), 120s for cloud providers
- **Embedding sanitization:** Progressive text length reduction on embedding failures
- **Langfuse tracing:** Every LLM call optionally traced with operation type, provider, model, input/output preview, latency, and error metadata

**Auto-escalation (Phase 6):**
The job evaluation system has a particularly clever pattern: it evaluates jobs locally first (fast, free), then automatically escalates to a cloud model if the local result looks suspicious (too few gaps identified, or rationale too brief). This gives you the cost savings of local inference with the quality backstop of cloud models.

```
Local Ollama evaluation
  → If gaps < 2 OR rationale < 20 words
    → Re-evaluate with Claude/GPT-4o/Gemini
    → Use cloud result instead
```

## SECTION 5: THE JOB SEARCH INTELLIGENCE SYSTEM (Phase 6)

### Technical Summary

The crown jewel of Recall.local is a complete job search intelligence platform that automatically discovers jobs, evaluates them against a resume using AI, and sends real-time alerts for high-fit positions.

**Discovery layer:**
- **JobSpy:** Scrapes Indeed, Glassdoor, ZipRecruiter (30 results/query, max 7 days old)
- **Adzuna API:** Structured job listings with salary data
- **SerpAPI:** Google Jobs aggregator
- **Direct ATS integrations:** Greenhouse, Ashby, Lever career page APIs
- **21 tracked companies** organized in 3 tiers with per-company title filters
- **Configurable search:** 11 target role titles, 4 location preferences, industry keywords
- **Scheduling:** n8n cron — 8-hour cycles for aggregators, 12-hour cycles for career pages

**Deduplication (3-tier):**
1. Exact URL match
2. Company + title + 7-day window match
3. Semantic similarity (0.92 cosine threshold) using job description embeddings
4. Fallback: difflib SequenceMatcher if Qdrant is unavailable

**AI evaluation pipeline:**
Every discovered job is evaluated against Jay's resume using a structured 5-dimension rubric:

| Dimension | Weight | What It Measures |
|-----------|--------|-----------------|
| Role Alignment | 32% | Title and core responsibilities match |
| Technical Alignment | 24% | Tools, systems, architecture demands |
| Domain Alignment | 16% | Industry and workflow context |
| Seniority Alignment | 14% | Scope and leadership level |
| Communication Alignment | 14% | Customer-facing and cross-functional needs |

**Scoring formula:**
```
Base score = weighted average of 5 dimensions (0-5 scale → 0-100)
+ Evidence bonus (up to +3 for matching skills with cited evidence)
- Gap penalties (critical: -10, moderate: -5, minor: -2, max -24 total)
= Final fit score (0-100)
```

**Evaluation output includes:**
- Fit score (0-100) with rationale
- Matching skills with evidence excerpts from resume
- Skill gaps categorized by severity (critical/moderate/minor) with learning recommendations
- Application tips and interview strategy
- Cover letter angle — the single strongest narrative thread

**Notification pipeline:**
- Telegram bot sends real-time alerts for jobs scoring 75+
- Location gating: only alerts for Remote or Twin Cities positions
- Message includes: title, company, fit score, URL

**Additional features:**
- Company profiler with tier management and profile refresh
- Gap aggregator that clusters skill gaps across all evaluated jobs and identifies top 3 learning priorities
- Cover letter drafter that generates context-aware drafts using matched skills and resume
- Resume ingestion with versioning and chunk-based storage for evaluation context

## SECTION 8: THE EVALUATION HARNESS

### Technical Summary

Recall.local has a production-grade evaluation system that continuously validates RAG quality and catches regressions.

**Test suites:**
- Core RAG eval cases (10-30 questions with expected document matches)
- Job search domain eval cases
- Learning domain eval cases
- Golden sets with ground truth for retrieval validation

**What gets tested:**
- **Citation validity:** Does every cited doc_id/chunk_id actually exist in the retrieved context?
- **Answer grounding:** Is the answer actually supported by the retrieved chunks?
- **Latency:** Is response time within acceptable bounds?
- **Pass rate trends:** Are we getting better or worse over time?

**Soak testing (Phase 4):**
- Configurable number of sequential runs
- Multiple suites per soak (core + job-search)
- Threshold enforcement: minimum 100% pass rate, max 15s average latency
- Trend artifacts: JSON + Markdown reports with pass rates, latency distributions, failure analysis

**Scheduled evaluation:**
- Daily cron-scheduled eval runs via `scheduled_eval.sh`
- Results stored in SQLite `eval_results` table
- Artifacts saved to `data/artifacts/evals/scheduled/`

**CI integration:**
- GitHub Actions workflow runs pytest on every PR
- Validates Python/shell syntax
- Smoke tests all major orchestration scripts

## SECTION 9: THE N8N WORKFLOW ORCHESTRATION

### Technical Summary

n8n is the automation backbone of Recall.local, handling scheduled pipelines, webhook routing, and multi-step orchestration.

**Key workflows:**

1. **Ingestion Webhooks:**
   - `POST /webhook/recall-ingest` → normalize → forward to bridge `/v1/ingestions`
   - `POST /webhook/recall-query` → forward to bridge `/v1/rag-queries`
   - `POST /webhook/recall-meeting` → forward to bridge `/v1/meeting-actions`

2. **Career Page Monitor (Phase 6B):**
   - 12-hour cron schedule
   - Iterates through 21 tracked companies
   - Calls Greenhouse/Ashby/Lever APIs with per-company title filters
   - 2-second delay between companies (rate limiting)
   - Routes new jobs through dedup → evaluation → notification pipeline

3. **Aggregator Polling (Phase 6B):**
   - 8-hour cron schedule
   - Rotates through search query batches (4 queries per cycle)
   - Calls JobSpy, Adzuna, SerpAPI with configured search parameters

4. **Evaluation + Notification (Phase 6C):**
   - Triggered after discovery runs
   - Evaluates new jobs against resume
   - Sends Telegram alerts for high-fit results

**Why n8n:**
- Visual workflow builder for non-code orchestration
- Built-in scheduling, retry, and error handling
- Self-hosted (runs in Docker alongside everything else)
- Webhook support for external integrations

## SECTION 12: OPERATIONS & INFRASTRUCTURE

### Technical Summary

Recall.local runs as a fully containerized stack with production-grade operational practices.

**Docker Compose stack (7 services):**
1. **Qdrant** (port 6333) — Vector database with persistent external volume
2. **Ollama** (port 11434) — Local LLM runtime with GPU passthrough
3. **n8n** (port 5678) — Workflow automation engine
4. **Recall Ingest Bridge** (port 8090) — FastAPI operations gateway
5. **Recall UI** (port 8170) — Main React dashboard (Nginx)
6. **Daily Dashboard** (port 3001) — Job search command center (Nginx)
7. **MkDocs** (port 8100) — Documentation server

**Operational practices:**
- **Health-aware startup:** Compose `depends_on` with `condition: service_healthy`
- **Deterministic restart ordering:** qdrant → ollama → bridge → n8n → mkdocs, with per-service health verification
- **Stack validation script:** Checks project labels, network attachment, volume mounts, container DNS resolution
- **Rate limiting:** In-memory fixed-window per-client (60s window, 120 requests default)
- **API authentication:** Optional X-API-Key header
- **Automated daily backups:** SQLite logical dumps + Qdrant full collection exports with SHA256 checksums, 14-day retention
- **Pre-flight checks:** Service connectivity validation before any operational changes
- **Langfuse tracing:** Optional observability for all LLM calls

**CI/CD:**
- GitHub Actions on every PR: Python syntax checks, shell script validation, unit tests, orchestration script smoke tests
- Soak testing with pass rate and latency thresholds
